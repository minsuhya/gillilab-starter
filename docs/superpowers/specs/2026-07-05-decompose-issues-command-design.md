# `/decompose-issues` 커맨드 설계

- 날짜: 2026-07-05
- 상태: 승인됨 (사용자 확인 완료)

## 배경 / 목표

큰 작업을 인자로 입력받아 독립적으로 처리 가능한 여러 GitHub 이슈로 분해하는 프로젝트 슬래시 커맨드를 만든다. 작업분석 → 작업분해 → 의존성분석 → 결과 출력 → GitHub 이슈 생성 여부 확인 → (승인 시) 순차 생성까지를 한 번의 커맨드 실행으로 처리한다.

## 범위

- 신규 파일 2개만 추가한다: 슬래시 커맨드 정의 1개, GitHub 이슈 템플릿 1개.
- 별도 서브에이전트, 워크플로우, 코드 변경은 필요 없다 — 커맨드 자체가 프롬프트 지시문이며, Claude Code 메인 대화에서 직접 실행된다.
- 대상 저장소는 GitHub CLI(`gh`)가 인증되어 있고 origin이 GitHub 저장소인 것을 전제로 한다.

## 구성 요소

### 1. 슬래시 커맨드 — `.claude/commands/decompose-issues.md`

Frontmatter:

```yaml
---
description: 작업을 독립적인 GitHub 이슈로 분해하는 전략 전문가
argument-hint: <분해할 작업 설명>
allowed-tools: Bash(gh label list:*), Bash(gh issue create:*), Bash(gh repo view:*), Bash(gh auth status:*)
---
```

동작 절차 (본문 지시문에 명시):

1. **작업분석** — `$ARGUMENTS`로 받은 작업의 목표/제약/범위를 파악한다. 범위가 모호하면 진행 전 짧게 되묻는다.
2. **작업분해** — 각자 독립적으로 완료 가능한 단위로 쪼갠다. 완전히 독립적이지 않은 단위는 의존성으로 명시한다.
3. **의존성분석** — 이슈 간 선후관계를 방향성 그래프로 정리한다 (`#2 depends_on #1` 형태). 순환 의존성이 발견되면 경고하고 분해를 재조정한다.
4. **분해된 이슈 출력** — 터미널에 표로 출력한다. 컬럼: 번호 / 제목 / 배경·목표 / 작업내용 / 완료조건(DoD) / 제안 레이블(area·complexity·type·priority) / 의존성.
5. **GitHub 이슈 생성 여부 질문** — `AskUserQuestion`으로 "지금 GitHub에 이슈로 생성할까요?" (예/아니오)를 묻는다.
6. **생성 (예 선택 시)** — 의존성 순서대로 `gh issue create`를 순차 실행한다. 먼저 생성된 이슈 번호를 이후 이슈 본문에 `Depends on: #N` / `Blocks: #N`으로 상호 참조한다. 4번에서 제안한 레이블을 `--label`로 부착한다.

엣지 케이스:

- `gh auth status`가 실패하면 생성 단계 진입 전에 확인하고, 실패 시 안내 메시지만 출력하고 생성을 중단한다.
- 레이블 제안 전 `gh label list`로 저장소에 실제 존재하는 레이블만 사용한다. 없는 레이블은 자동 생성하지 않고 생략한다.
- 순환 의존성이 발견되면 4번 출력 전에 경고하고 사용자에게 재분해 여부를 확인한다.

### 2. 이슈 템플릿 — `.github/ISSUE_TEMPLATE/task.md`

단일 범용 템플릿. 커맨드가 `gh issue create --body`로 생성하는 본문 구조와 1:1 대응한다 (필드가 같아야 사람이 수동으로 여는 이슈와 커맨드가 만든 이슈가 일관된 형태를 가짐).

필드: 배경/목표, 작업 내용(체크리스트), 완료 조건(DoD), 의존성, 참고.

## 연동 방식

- 레이블 제안은 `gh label list`로 저장소의 실제 레이블(area:/complexity:/type:/priority:)을 조회해 매칭하므로, 레이블 스키마가 바뀌어도 커맨드를 수정할 필요가 없다.
- 템플릿 파일은 커맨드가 코드로 참조하지 않는다 — 사람이 수동으로 이슈를 열 때 동일한 구조를 보장하기 위한 문서적 장치다.

## 테스트 방법

- 샘플 작업 설명으로 커맨드를 직접 실행해 1~4단계 출력이 의도한 표 형식으로 나오는지 확인한다.
- "아니오"를 선택했을 때 GitHub에 아무 이슈도 생성되지 않는지 확인한다.
- "예"를 선택했을 때 의존성 순서대로 이슈가 생성되고 본문에 상호 참조와 레이블이 정확히 반영되는지 `gh issue view`로 확인한다.
- 순환 의존성을 유도하는 입력으로 경고 동작을 확인한다.

## 비범위 (Out of scope)

- type별 다중 템플릿(feature/bug/docs/test)은 만들지 않는다 — 단일 범용 템플릿으로 통일.
- 상위 트래킹 이슈 + 서브태스크 체크리스트 방식은 사용하지 않는다 — 개별 이슈 + 본문 상호참조 방식만 사용.
- 레이블이 저장소에 없을 경우 자동 생성하는 기능은 만들지 않는다.
